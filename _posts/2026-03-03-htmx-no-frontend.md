POV: вы бэкендер и хотите админку для сервиса. Вызывать методы через Swagger в 2026 году уже странно.

Обычно начинается так:

- `npm`
- `npx`
- `tailwind`
- `nodejs`
- `react/vue/angular/solid`
- десятки конфигов, которые быстро устаревают

Для простых CRUD-панелей это часто избыточно.

## Почему так сложно

Классический путь данных обычно выглядит так:

SQL или ORM
→ backend-сервис
→ REST handler
→ JSON
→ сеть
→ десериализация
→ state
→ props
→ компонент
→ HTML

Много слоёв ради обычной таблицы.

## Идея HTMX

HTMX предлагает в ряде сценариев отказаться от JSON на клиенте и возвращать сразу HTML.

Пример:

```html
<button hx-get="/users"
        hx-target="#table"
        hx-swap="innerHTML">
  Загрузить
</button>
```

Что происходит:

- по нажатию уходит GET-запрос на `/users`
- сервер возвращает HTML
- HTML вставляется в `#table`, заменяя содержимое

Тогда путь данных проще:

SQL или ORM
→ backend
→ HTML

Иногда лучший фронтенд — это отсутствие отдельного фронтенда.

## Подключение

HTMX — это `0-build` подход:

```html
<script src="https://unpkg.com/htmx.org@1.9.12"></script>
```

Без `npm`, `vite`, `webpack` и прочей инфраструктуры.

## Когда стоит использовать

- админки и внутренние панели
- дашборды
- MVP, где важнее функциональность, чем сложный UI

## Когда не стоит

- сложный клиентский state
- realtime с тонкой синхронизацией
- canvas и графические редакторы
- большие SPA с автономной логикой на клиенте
- проекты с уже сформированной frontend-командой

У каждой технологии есть свой контекст. HTMX не замена всему, но для серверно-ориентированных интерфейсов это может сильно сократить время и сложность.
