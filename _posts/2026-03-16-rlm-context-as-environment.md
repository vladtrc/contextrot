Чем больше токенов в контексте, тем хуже модель рассуждает над каждым из них. Фундаментальное свойство attention, общее для всех моделей. На 1M токенах сигнал тонет в шуме.

RAG казался ответом: не грузи всё в контекст, положи в векторную базу и доставай нужное. Работает, но только если "нужное" это несколько чанков по ключевым словам. Если ответ требует агрегации по всему документу или понимания глобального контекста — RAG ломается(

Идея RLM: **использовать контекст как внешнюю среду, а не место куда надо всё запихнуть**.

Весь длинный контекст хранится как переменная в персистентном Python (или каком-то другом) REPL. Модель его не видит. Она пишет код чтобы с ним взаимодействовать: читает кусками, строит индексы, делает срезы. Как разработчик с SQL базой: смотришь на данные через запросы, а не грузишь всю базу в память.

_Задача: найти паттерн ошибок в JSON-логе на 50MB (~10M токенов, 2.8M событий)._

| Шаг | Обычный агент (read_file) — что в контексте | Токены | RLM — что в контексте | Токены |
|---|---|---|---|---|
| Системный промпт | `"You are a helpful assistant..."` | 50 | `"Your context: 52 428 800 chars. Available: context, llm_query(), FINAL()"` | 70 |
| Запрос | `"Найди паттерн ошибок в логах"` | 60 | `"Найди паттерн ошибок в логах"` | 80 |
| Чтение файла | Tool call `read_file("app.log")` → `{"level":"error","msg":"timeout","svc":"auth"...` _(середина JSON-объекта, структура сломана)_ | 560 | `>>> import json; data = json.loads(context)` → `[stdout: 31 chars · "Loaded: 2 847 291 events"]` | 110 |
| Агрегация | ещё вызовы `read_file` с разными офсетами — фрагменты без связи, агрегировать нельзя | 1060 | `>>> errors = [e for e in data if e['level']=='error']; print(len(errors), errors[0])` → `[stdout: 61 chars · "18 442 errors · {'ts':'2026-01-14T03:12:01','svc':'auth'..."]` | 160 |
| Группировка по сервису | _(нет единого представления данных — файл читается кусками)_ | — | `>>> from collections import Counter; print(Counter(e['svc'] for e in errors).most_common(5))` → `[stdout: 78 chars · "[('auth', 9821), ('payments', 5302)...]"]` | 210 |

Рекурсия: корневая модель (оркестратор, который управляет всем процессом) замечает интересный раздел → запускает суб-вызов с этим разделом в изолированном контексте → суб-вызов делает плотный анализ → возвращает только компактный результат → корневая модель продолжает. Промежуточный контекст суб-вызова выбрасывается. Корень оркестрирует, не читает. Если в контексте лежат структурированные данные — JSON, CSV, XML — модель может распарсить их питоном и делать с ними что угодно: фильтровать, агрегировать, трансформировать. Точечный поиск по всему объёму данных — один вызов `string.find()`.

BrowseComp+ (6–11M токенов):
- RAG + BM25: **51%**
- Summary agent: **70.5%**
- RLM (GPT-5 корень + GPT-5 mini суб-вызовы): **91.3%**, стоимость **$0.99**

ULong Pairs (информационно плотный бенч):
- Чистый GPT-5: **<0.1%**
- RLM с GPT-5: **58%**

## Claude Code же уже так делает?

Первая реакция на эту информацию: "это просто субагенты, мы так уже делаем.. мой клодкод всё это умеет...". И частично это справедливо — паттерн не новый и результаты похожи. Но есть три конкретных отличия которые важны.

В стандартном агентном скаффолде — Claude Code, LangGraph, любой другой — результат каждого суб-вызова **дописывается токенами в контекст оркестратора**. Суб-агент прочитал документ и вернул резюме — это резюме теперь в истории. Следующий вызов — ещё. Контекст оркестратора растёт с каждым шагом. Сам документ тоже читается через инструменты — `read_file` возвращает текст в историю.

В RLM иначе:

**Контент никогда не становится токенами.** Документ на 6M токенов — Python-переменная в REPL. Модель пишет код чтобы с ним взаимодействовать (`doc[100000:150000]`, `index.search(query)`). Не tool call который возвращает текст — код который исполняется, результат которого маленький return value.

**Промежуточный контекст суб-вызовов выбрасывается.** Субагент думал 30К токенов — оркестратор видит только компактный return. В обычном агенте весь ход мышления оседает в истории.

**Рекурсия происходит внутри кода, не в истории сообщений.** Вызов агента в Claude Code — событие в диалоге, оно логируется. В RLM рекурсия — Python-функция в REPL. С точки зрения оркестратора это результат выражения.

Итог: оркестратор Claude Code при обработке большого документа упрётся в своё контекстное окно. Оркестратор RLM при 10M токенах держит в контексте только текущий код плюс компактные результаты завершённых суб-вызовов. Контекст не растёт пропорционально размеру входа.

Авторы в статьях разграничивают: агент — это автономия, пространство действий, цикл наблюдения. RLM — это REPL для управления длинным контекстом.

В общем, RLM может быть внутри агента, но текущие агенты != RLM.

## Почему этого нет в продакшне

Нам рассказывают что в Anthropic уже кодят агенты и решают любые задачки, время между теорией и продом сокращается в разы. Так почему ж мы этого не видим?

Вышел [независимый reproduction paper](https://arxiv.org/abs/2603.02615) ("Think, But Don't Overthink") и несколько практических тестов.

**Разброс результатов убивает надёжность.** 30 идентичных запусков — 10–17% критических отказов. В боевых условиях неприемлемо. Авторам пришлось добавить тройной перезапуск с выбором лучшего результата чтобы получить 100% — что удваивает стоимость.

**Глубина рекурсии > 1 ломает всё.** Kimi K2, который сам по себе делает 86.6% на задачах с длинным контекстом, под RLM depth=1 (корневая модель вызывает суб-агента — один уровень) падает до 60%, при depth=2 (суб-агент сам запускает ещё суб-агента — два уровня вложенности) — до 55%. RLM сделал сильную модель хуже. DeepSeek деградирует аналогично. Теоретическая красота истинной рекурсии разбивается о то что модели в ней не умеют надёжно работать(( ну по крайней мере ПОКА что не умеют

**Три задокументированных сценария отказа:** параметрическая галлюцинация (модель игнорирует контекст и отвечает из обучающих весов), потеря структуры вывода (возвращает сырой Python вместо ответа), бесконечные циклы псевдоверификации (без финального ответа). Задержка при двух уровнях рекурсии вырастает с 3.6 секунды до 344 секунд, абсолютная руина

**Работает только на топовых моделях.** GPT-4o решил ноль задач из шести на агрегации — задачах где нужно обойти весь документ целиком и что-то посчитать или собрать: "сколько раз встречается X", "в каком сервисе больше всего ошибок". Модель должна сама написать корректный Python для неизвестной структуры данных — слабые модели на этом ломаются.

Помимо теоретической есть инфраструктурная сложность: постоянная сессия Python REPL с доступом к данным пользователя — это переписывать весь текущий sandbox, совсем другой мониторинг и точки отказа. Стандартная инфраструктура не хранит состояние (писал об этом в прошлых постах); RLM требует живую сессию на всё время выполнения задачи.

Будет ли в эксплуатации? Вероятнее всего, да. Теория вышла в декабре 2025, воспроизводящая работа — в марте 2026. Цикл от "интересный препринт" до "в работе" обычно год+. Сценарии отказов задокументированы — значит их можно устранять.

RAG быстрее и дешевле там где нужна скорость и точечный поиск. Но RAG — это эвристика, не инструмент рассуждения. Вероятно мы увидим чтото типа: RLM как оркестратор, векторная база как один из инструментов внутри. RLM решает когда и что искать, RAG делает быстрый поиск по нужному куску, RLM интерпретирует результаты и решает что дальше.
