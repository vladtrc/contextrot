В декабре 2025 вышел препринт [Recursive Language Models](https://arxiv.org/abs/2512.24601) (MIT, Zhang, Kraska, Khattab). Шумиха была большая — "смерть RAG", "конец контекстного окна". Я сначала не понял хайп. Потом перечитал.

## Что такое RLM

Про [context rot]({% link _posts/2026-02-28-context-rot.md %}) уже писал. Если коротко: чем больше токенов в контексте, тем хуже модель рассуждает над каждым из них. Это не баг конкретной модели — это фундаментальное свойство attention. На 1M токенах сигнал тонет в шуме.

RAG казался ответом: не грузи всё в контекст, положи в векторную базу и доставай нужное. Работает — но только если "нужное" это несколько чанков по ключевым словам. Если ответ требует агрегации по всему документу или понимания глобального контекста — RAG ломается. Это поисковик приклеенный к reasoning engine скотчем.

Идея RLM простая и при этом неожиданно точная: **контекст — это внешняя среда, а не место куда надо всё запихнуть**.

Технически: весь длинный контекст хранится как переменная в персистентном Python REPL. Модель его не видит напрямую. Вместо этого она пишет код чтобы с ним взаимодействовать — читает кусками, строит индексы, делает срезы. Как разработчик с SQL базой: смотришь на данные через запросы, а не грузишь всю базу в память.

Дальше — рекурсия. Корневая модель замечает интересный раздел → запускает суб-вызов с этим разделом в изолированном контексте → суб-вызов делает плотный анализ → возвращает только компактный результат → корневая модель продолжает. Промежуточный контекст суб-вызова полностью выбрасывается. Корень никогда не тонет в промежуточных выводах — он оркестрирует, а не читает.

Числа. BrowseComp+ (6–11M токенов):
- RAG + BM25: **51%**
- Summary agent: **70.5%**
- RLM (GPT-5 корень + GPT-5 mini суб-вызовы): **91.3%**, стоимость **$0.99**

ULong Pairs (информационно плотный бенч):
- Чистый GPT-5: **<0.1%**
- RLM с GPT-5: **58%**

Это не +5 процентных пунктов. Задачи которые вообще не решались — теперь решаются.

## Но подождите — Claude Code же уже так делает?

Первая реакция HackerNews на бумагу: "это просто субагенты, мы так уже делаем". И частично это справедливо — паттерн не новый. Но есть три конкретных отличия которые важны.

В стандартном агентном скаффолде — Claude Code, LangGraph, любой другой — результат каждого суб-вызова **аппендится токенами в контекст оркестратора**. Суб-агент прочитал документ и вернул резюме — это резюме теперь в истории. Следующий вызов — ещё. Контекст оркестратора растёт с каждым шагом. Сам документ тоже читается через инструменты — `read_file` возвращает текст в историю.

В RLM иначе:

**Контент никогда не становится токенами.** Документ на 6M токенов — Python-переменная в REPL. Модель пишет код чтобы с ним взаимодействовать (`doc[100000:150000]`, `index.search(query)`). Не tool call который возвращает текст — код который исполняется, результат которого маленький return value.

**Промежуточный контекст суб-вызовов выбрасывается полностью.** Субагент думал 30К токенов — оркестратор видит только компактный return. В обычном агенте весь ход мышления оседает в истории.

**Рекурсия происходит внутри кода, не в истории сообщений.** Вызов агента в Claude Code — событие в диалоге, оно логируется. В RLM рекурсия — Python-функция в REPL. С точки зрения оркестратора это просто результат выражения.

Итог: оркестратор Claude Code при обработке большого документа упрётся в своё контекстное окно. Оркестратор RLM при 10M токенах держит в контексте только текущий код плюс компактные результаты завершённых суб-вызовов. Контекст не растёт пропорционально размеру входа.

Авторы сами аккуратно разграничивают: агент — это автономия, пространство действий, цикл наблюдения. RLM уже, конкретнее: скаффолд для управления длинным контекстом. RLM может быть внутри агента. Но это не синонимы.

## Почему этого нет в продакшне — и будет ли

Вышел [независимый reproduction paper](https://arxiv.org/abs/2603.02615) ("Think, But Don't Overthink") и несколько практических тестов. Данные интересные.

**Variance убивает надёжность.** 30 идентичных запусков — 10–17% catastrophic failure rate. Для продакшн пайплайна неприемлемо. Авторам пришлось добавить best-of-3 retry чтобы получить 100% — что удваивает стоимость.

**Глубина рекурсии > 1 ломает всё.** Kimi K2, который нативно делает 86.6% на long-context задачах, под RLM depth=1 падает до 60%, при depth=2 — до 55%. RLM сделал сильную модель хуже. DeepSeek деградирует аналогично. Теоретическая красота истинной рекурсии разбивается о то что модели в ней не умеют надёжно работать.

**Три задокументированных failure mode:** параметрическая галлюцинация (модель игнорирует контекст и отвечает из обучающих весов), formatting collapse (возвращает сырой Python вместо ответа), performative reasoning loops (бесконечные "верификации" без финального ответа). Latency при depth=2 вырастает с 3.6 секунды до 344.

**Работает только на frontier reasoning моделях.** GPT-4o — 0/6 на aggregation tasks. Только топ, только дорого.

Плюс инфраструктурная сложность: persistent Python REPL с доступом к данным пользователя — это другой деплой, другой sandbox, другой мониторинг. Стандартная inference инфраструктура stateless; RLM требует живую сессию на всё время выполнения задачи.

Будет ли в проде? Скорее да. Бумага вышла в декабре 2025, reproduction — в марте 2026. Цикл от "интересный препринт" до "в проде" обычно год+. Failure modes задокументированы — значит их можно фиксить. Первыми это сделают там где боль ощутимая и платить готовы: юридический анализ больших документов, финансовые отчёты, code review на крупных репозиториях.

RAG мёртв? Нет. RAG быстрее и дешевле там где нужна скорость и точечный поиск. Но RAG — это эвристика, не reasoning tool. Интересное будущее: RLM как оркестратор, векторная база как один из инструментов внутри. RLM решает когда и что искать, RAG делает быстрый поиск по нужному куску.
